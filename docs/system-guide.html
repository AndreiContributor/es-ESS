System design and deployment guide

PV-surplus EV charging when a battery ESS changes the power flow

A generic, adaptable explanation of how a three-phase Fronius PV system, Fronius Wattpilot, Victron inverter/chargers, a GX device and a lithium battery bank can be coordinated by es-ESS—without confusing battery-supported power with real PV surplus.

01 · The system

Generic architecture, concrete example

The concepts apply to many AC-coupled three-phase systems. Product names below are examples, not universal requirements. Electrical ratings, protection, phase allocation and battery limits must always be verified for the actual installation.

PV generation

Three-phase grid-tied inverter

Produces AC power from the PV array and exposes production data over the local network.

Example: Fronius Symo 10.0-3-M with 24 panels.

Point of connection

Bidirectional smart meter

Measures import and export at the grid connection and communicates with the PV inverter.

Example: Fronius Smart Meter 63 A via Modbus RTU.

Energy storage

Three-phase battery ESS

One inverter/charger per phase, coordinated by a GX device and a managed lithium battery bank.

Example: 3 × MultiPlus-II 48/4k5/55-32 and 2 × 16 kWh LiFePO₄.

Flexible demand

EV charging

A networked wallbox whose current and phase mode can be controlled in small steps.

Example: Fronius Wattpilot Go 22 kW.

Units matter: inverter power is measured in kW, battery energy in kWh, and current in A. “10.0” in a Symo model name is an inverter power class; two 16 kWh batteries provide about 32 kWh nominal energy, not 32 kW.

02 · Architecture

Power flows and information flows are different

A reliable controller needs more than the power seen by one meter. The solid lines below are electrical paths; dashed lines are measurement or command paths.

Example placement: Wattpilot Position=0

es-ESS publishes 0 as AC-out and 1 as AC-in. This is a configured reporting value, not automatic proof of the physical wiring. Check the deployed /data/es-ESS/config.ini, the Cerbo device view and the electrical diagram.

Grid failure is a separate design case

House backup can remain stable while EV Auto/Eco charging stops. Whether an AC-coupled PV inverter can continue during an outage depends on its AC location, certified anti-islanding behavior, inverter/charger sizing and Victron’s AC-coupling rules. This is installer-level design, not an es-ESS setting.

03 · Why the problem appears

A meter sees net flow—not the history of each watt

Before storage, grid export is a good approximation of unused PV. After storage, a battery inverter can actively change that grid flow. The EV controller must therefore know the battery contribution.

Before batteries

Export usually means unused PV

If PV produces 5 kW and the house consumes 2 kW, roughly 3 kW reaches the grid point. A surplus controller can safely allocate most of that to the EV.

After batteries

ESS holds the grid target

When the EV demand rises or PV falls, the Victron ESS may discharge the battery so the grid remains near 0 W or a small configured export.

Control-loop trap

The shortfall becomes invisible

A controller that sees only the Fronius side can conclude that charging is still covered, even though the Victron battery is now supplying part of the car.

Worked example: apparent surplus versus real overhead

Power itemBefore cloudAfter cloud, ESS compensates
PV generation6,000 W2,700 W
House excluding EV1,500 W1,500 W
EV charge4,140 W (3 × 6 A)4,140 W requested
BatteryCharging or idleAbout 2,940 W discharge
Grid meterSmall exportStill near its target
Real PV available to EVAbout 4,500 WAbout 1,200 W

Key lesson: a near-zero grid value does not mean “the EV is running on PV.” It means the complete site is balanced at the meter. PV, battery and grid contributions must be evaluated together.

What the supplied system screen demonstrates

A representative screen can simultaneously show PV at 0 W, AC loads around 1.1 kW, battery discharge around 1.26 kW, and grid flow of only a few watts. That is normal ESS regulation: the battery supplies the loads while the grid stays close to its target. The screen’s Grid setpoint 0 W is a different setting from the example -50 W configuration described later.

04 · es-ESS control model

Calculate real overhead, reserve it, then grant an allowance

es-ESS uses the GX system view because it includes battery and grid behavior. SolarOverheadDistributor reconstructs the energy budget and assigns the Wattpilot only the power it is allowed to consume.

raw overhead = max(0, grid feed-in + controlled-consumer power + battery power)
with the repository convention: battery charging is positive; battery discharge is negative

consumer allowance = raw overhead distributed by priority and battery-charge reservation
scripted consumers use minimum/step increments; explicit HTTP/MQTT on/off loads receive their complete request or zero

1 · Observe

Require fresh inputs

Read all three grid phases, battery power/SOC, current consumer demand and Wattpilot state. Missing critical inputs produce zero allowance or a fail-safe stop.

2 · Allocate

Reserve and distribute

Keep the configured battery-charge reservation, then allocate by priority. Scripted consumers use minimum/step increments; explicitly configured HTTP/MQTT on/off loads are granted their complete request or zero.

3 · Command

Control only Auto/Eco

Start, stop, change current or switch phases only when Wattpilot mode telemetry confirms ECO control. Manual remains owned by the user.

Atomic on/off loads: es-ESS controls a pool pump, heater, Shelly device, or similar load only when an operator explicitly adds an HttpConsumer:* or MqttConsumer:* section. Set Request to the device's complete operating requirement; legacy Minimum and StepSize values do not apply to these NPC consumers. If 800 W is available, a higher-priority 1000 W pump is skipped and an eligible lower-priority 500 W heater may receive 500 W. This prevents unusable partial reservations. Verify the device's on/off endpoint, status, power reporting, and safe fallback before enabling automatic control.

Rare protocol charging reasons: the upstream Wattpilot firmware API names modelStatus values 8–11 and 13–14 as ChargingBecause... states. es-ESS therefore treats them as active charging only after command-authority, fresh-telemetry, no-grid, phase-transition and confirmed-connection safety gates have passed. In Manual mode they remain reporting-only and never authorize es-ESS charging commands.

Entry, changes between these values and exit are logged once at INFO with the stable marker Wattpilot special charging model status; an unchanged state is not repeated every control cycle. Natural occurrences can be found in long-running production logs with:

grep -F "Wattpilot special charging model status" /data/log/es-ESS/current.log

06 · Battery policy

Strict PV-only and bounded cloud bridging are different policies

Both profiles prevent a battery-assisted start. The difference is what happens to an already-running charge when a short cloud suddenly removes some PV.

Alternative

Strict PV-only

The controller reduces current, moves to one phase or stops once the configured allowance and safety grace are exhausted. It minimizes home-battery cycling but may cause more EV interruptions.

# Never bridge a PV dip with the home battery
BatteryAssistEnabled=false
AllowGridCharging=false

Recommended example

Bounded battery assist

A running charge may survive a short cloud when SOC is valid, selected-battery activity is recent, and shortfall and time remain within limits. Venus OS does not periodically republish an unchanged SOC value, so finite /Dc/Battery/Power updates provide the heartbeat used to trust the cached SOC. Missing or invalid SOC, or a missing, invalid or stale heartbeat, fails closed. The maintained 1 kW shortfall default smooths small dips while making current reduction, phase-down or stop happen earlier when the EV would otherwise lean hard on the home battery.

BatteryAssistEnabled=true
BatteryAssistSocMin=50
BatteryAssistMaxSeconds=600
BatteryAssistMaxShortfallW=1000
BatterySocFreshSeconds=15
BatteryAssistRecoverySeconds=120
AllowGridCharging=false

What the recommended example permits

  • Only an already-running Auto/Eco session can receive assistance.
  • SOC must be finite and remain at or above 50%; selected-battery /Dc/Battery/Power activity must be no older than 15 seconds because it is the liveness heartbeat for cached SOC.
  • The calculated PV shortfall must not exceed 1 kW with the maintained daily-use default.
  • One assist window lasts no more than 600 seconds.
  • PV must fully recover for 120 seconds before another locked-out assist window becomes available.
  • Assistance cannot start an EV session and cannot justify switching from one phase to three.
  • With AllowGridCharging=false, sustained grid import still invokes the stop guard.

Energy scale: 1 kW for 600 seconds is at most about 0.17 kWh before conversion losses—around 0.5% of a 32 kWh nominal battery bank. The actual amount is usually lower because assistance tracks only the shortfall. Raising this value makes the EV tolerate larger PV dips but also allows more home-battery discharge before the controller reduces current, phases down or stops.

EvPriorityOverBatteryCharge=true is not battery discharge assistance. Above EvPriorityMinSoc, it allows real PV that would otherwise charge the stationary battery to be redirected to the connected EV. This bypass and battery assist both require valid SOC plus a finite selected-battery /Dc/Battery/Power update within BatterySocFreshSeconds; missing or invalid SOC, or missing, invalid or stale battery activity, disables them for that cycle. The controller still requires an actual distributor allowance for a new start.

07 · Grid setpoint ownership

DefaultPowerSetPoint=-50 actively changes Victron ESS

This setting is global to es-ESS. It is not merely a documentation default and it is not part of the Wattpilot allowance calculation.

Baseline

Negative exports

-50 asks Victron ESS to export approximately 50 W. A positive value asks for grid import. The physical result will move around the target because meters and control loops have delays.

Runtime behavior

es-ESS writes the setting

At startup, es-ESS publishes the calculated value through local Venus MQTT to /Settings/CGwacs/AcPowerSetPoint. It restores the baseline after a managed error or graceful shutdown.

Energy source

The target is source-blind

The 50 W export can be supplied by PV or by battery discharge. The setpoint does not prove PV surplus; this is why the overhead calculation separately includes battery power.

combined grid setpoint = clamp(DefaultPowerSetPoint + active service adjustments, GridSetPointMinW, GridSetPointMaxW)

Example with NoBatToEV active and a commissioned maximum of at least +1,950 W:
−50 W baseline + 2,000 W request = +1,950 W import target

Scope and interaction with services

  • If NoBatToEV=false, no currently active service adds a grid-setpoint request, so es-ESS normally calculates -50 W.
  • NoBatToEV can add a positive request to offload the EV shortfall to the grid, preventing that EV portion from draining the battery.
  • The maintained configuration defaults both bounds to the existing baseline. Dynamic adjustments therefore remain clamped to that baseline until the operator enters site-approved limits; each distinct clamp is logged.
  • That behavior conflicts with a no-grid Auto/Eco objective, so keep NoBatToEV=false with the recommended Wattpilot battery-assist profile.
  • The dormant ChargeCurrentReducer has a separate setting of the same name in its own section; it is not an active service in this checkout.
  • es-ESS republishes the grid setpoint when its calculated value changes. It does not necessarily overwrite every later external change on each five-second management cycle.
  • A 50 W export is far below the usual EV minimum of about 1.38 kW, so it cannot alone support a normal 6 A start. It can still contribute to a misleading source-blind native measurement.

Configuration-file distinction: config.sample.ini is a template. The deployed service reads /data/es-ESS/config.ini. Editing the sample does not modify an already-installed system unless the production file is updated too.

Battery reservation

MinBatteryCharge is a desired PV reservation for stationary-battery charging, evaluated from SOC. It is not a BMS current limit. With the example curve, reservation is high at lower SOC and approaches 250 W near 99%.

If EvPriorityOverBatteryCharge=true, SOC is high enough, and selected-battery activity is recent, a connected Auto/Eco Wattpilot may ignore that reservation to take real PV. Missing or invalid SOC, or a missing, invalid or older-than-BatterySocFreshSeconds battery-power heartbeat, disables the bypass. It still cannot start from battery power.

Current and phase thresholds

The Wattpilot family commonly starts around 6 A: approximately 1.38 kW on one phase and 4.14 kW on three phases at nominal voltage. Use hysteresis, so the three-phase start threshold is higher than the stop threshold.

Do not infer a safe 32 A setting from a 22 kW product label. The lowest limit among supply, protection, cabling, adapter, wallbox, vehicle and local rules wins.

Worked behavior example

One-phase start, three-phase operation and a cloudy PV dip

This timeline uses the parameter values in the recommended adaptable starting point above. Wattage values are approximate because the electrical minimum follows live phase voltage. The fresh SolarOverheadDistributor allowance assigned to Wattpilot—not total PV generation or raw grid export—is authoritative.

  1. Below the one-phase minimum: wait.
    At 6 A and about 230 V, one-phase charging needs approximately 1.38 kW of assigned PV. Below that level Auto/Eco remains stopped. Battery assist cannot create a new charging session.
  2. Hold enough PV for 60 seconds: start on one or three phases.
    When fresh assigned allowance meets the electrical minimum for MinOnOffSeconds=60, charging starts at a bounded current. It starts on one phase below ThreePhasePvSurplusStartW=4500, or directly on three phases when at least 4,500 W is already assigned. This direct start matches normal Wattpilot behavior and does not use the running-session 1-to-3 timer.
  3. If a cloud passes on one phase: bridge, reduce or stop.
    For an already-running charge, eligible battery assist may hold the existing one-phase current for up to BatteryAssistMaxSeconds=600 when SOC is at least 50%, selected-battery power activity is no older than BatterySocFreshSeconds=15, shortfall is no greater than 1,000 W, grid import remains within the safety limit, and telemetry is fresh. Missing or invalid SOC, or a missing, invalid or stale battery heartbeat, clears/refuses assist. If PV recovers, assist clears and one-phase charging continues. If the deficit persists but assigned PV still supports approximately 1.38 kW, charging reduces to a PV-supported current; below that minimum it stops after AllowanceDropGraceSeconds=30. An assist timeout remains locked out until PV fully covers EV demand for 120 seconds or the car disconnects. A later restart again requires sufficient assigned PV for 60 seconds.
  4. While already charging on one phase, hold 4,500 W for 10 minutes: become eligible for three phases.
    Once assigned allowance reaches ThreePhasePvSurplusStartW=4500, the shared MinPhaseSwitchSeconds=600 timer begins. On the normal adjustment path, a dip shorter than SurplusDropGraceSeconds=30 preserves the candidate only above the effective three-phase floor; a deeper or longer normally evaluated dip resets it. If bounded battery assist is actively holding the existing one-phase charge, an already-started candidate timer is intentionally left running even through a deeper dip, potentially for up to BatteryAssistMaxSeconds=600. Assist cannot create the candidate or switch phases. Fresh assigned allowance must recover to the full 4,500 W before the controller can command and confirm three-phase mode. The 4,500 W default remains above the normal three-phase 6 A floor while making phase-up less conservative than a 5,000 W threshold.
  5. With sufficient PV: stay on three phases.
    There is no maximum three-phase duration. Charging can remain in that mode indefinitely while assigned PV stays above the effective phase-down threshold.
  6. A cloud lowers PV: debounce a brief allowance drop, then bridge when safe.
    The configured stop threshold is ThreePhasePvSurplusStopW=4100, while the effective threshold also respects the live three-phase electrical minimum. If the distributor cannot satisfy that atomic request, it truthfully publishes 0 W. An already-running charge retains its current command for up to AllowanceDropGraceSeconds=30; recovery clears the timer, while a sustained deficit proceeds to phase-down or stop. Stale grid telemetry and the grid-import guard are never delayed. Eligible battery assist may bridge a longer shortfall only within the configured SOC, battery-activity, power and duration bounds.
  7. PV recovers during the bridge: remain on three phases.
    The phase-down candidate is cancelled when assigned PV again supports three phases. Full PV coverage for BatteryAssistRecoverySeconds=120 satisfies the configured assist-recovery condition.
  8. The deficit persists: reduce phase or stop.
    After the shared phase timer, the controller switches to one phase when fresh allowance can support it; otherwise it stops. If SOC falls below 50%, the shortfall exceeds 1,000 W, assist times out, or other allowance conditions make the long bridge ineligible, phase reduction or stop may occur after the 30-second allowance-drop grace and before the normal 10-minute wait completes. Stale grid telemetry and sustained grid import retain their earlier safety response.
  9. Later phase-up normally requires stability again.
    After a completed phase-down, a new 1-to-3 candidate again uses the shared 600-second stability/cooldown setting. During an eligible one-phase battery bridge, however, an existing candidate may mature while PV is low; if fresh allowance then recovers to at least 4,500 W, the controller may switch without starting another complete 600-second interval. The full threshold at the command boundary prevents a battery-only phase-up.

Energy-source boundary: with AllowGridCharging=false, es-ESS does not intentionally use grid power as fallback. During an eligible bridge, Victron ESS determines the physical battery/grid power flow; Wattpilot control only decides whether the already-running EV session may continue.

Fake battery monitors: SolarOverheadDistributor exposes reservation/consumer information as battery-like D-Bus services for VRM visibility. Manually select the actual managed battery/BMS under GX System Setup → Battery Monitor. Do not leave automatic selection where a fake monitor could be chosen.

09 · Deployment

Install, configure one service at a time, then observe

Validated runtime lock: this checkout starts only on the clean Venus OS release v3.75. Qualified builds and all other versions fail closed before services, MQTT or grid-setpoint writes begin. The v3.75 upgrade, idle/no-vehicle, Manual charging, Manual current-change, Manual recovery and supervised Auto/Eco PV-surplus daylight checks have passed on build 20260624163305. Wattpilot control remains blocked until charger firmware 42.5 is received through fwv telemetry.

  1. Confirm the electrical system first.
    Record PV and Wattpilot AC positions, Smart Meter direction, grid sign, phase assignment, protection, battery BMS limits, and whether AC-coupled PV is intended to operate during an outage.
  2. Prepare the GX device.
    Confirm the supported clean Venus OS release (v3.75), enable SSH/root access, ensure the local MQTT broker is available, and verify that the Fronius inverter, grid phases, battery and MultiPlus system have fresh values in the GX interface.
  3. Stabilize networking.
    Put the Cerbo GX, Fronius inverter and Wattpilot on a reliable LAN. Use DHCP reservations or static addresses, and confirm the Cerbo discovers the Fronius inverter in the correct AC position.
  4. Back up before installation.
    Save the current GX/ESS settings and any existing /data/es-ESS/config.ini. Record the original Victron grid setpoint because es-ESS will write the configured baseline.
  5. Download and install es-ESS.
    Run the repository installation commands on the GX device. The installer creates the service link, adds a recovery entry to /data/rc.local, and copies config.sample.ini to config.ini only when no production config exists.
  6. Install the Wattpilot Python dependency.
    Install python3-pip when needed and then websocket-client. Venus OS upgrades can remove extra packages, so recheck this dependency after firmware updates.
  7. Configure and restart.
    Edit /data/es-ESS/config.ini, enable SolarOverheadDistributor first, validate its calculations, then enable FroniusWattpilot and restart gracefully.
  8. Commission with low risk.
    Start with the vehicle disconnected, conservative current limits, Use PV surplus off, flexible tariff off, and supervision at the site. Select Auto through the VRM web/Remote Console EVCS mode control or, on Android, press right once from Manual in the dedicated home-screen VRM EV Charging Station widget. Do not confuse it with the informational EVCS area inside the VRM app installation schematic. Connect the vehicle only after the authority diagnostic is healthy. Validate one-phase operation before allowing the three-phase threshold to be reached.

Read-only es-ESS daily report

Use the daily report for historical/end-of-day validation; the health monitor above remains the live snapshot tool. The analyzer always requires [Common] LogLevel=APP_DEBUG (or more verbose). Yesterday and older dates require a complete requested window. --date today analyzes diagnostic records available from Venus-local midnight through execution and remains INCOMPLETE unless it already proves an ANOMALY. A bounded, exact, read-only query to com.victronenergy.settings /Settings/System/TimeZone makes the report calendar agree with logging even when the shell runs in UTC. It prints the report timezone, requested period, analysis cutoff, first/last evidence, evidence duration, elapsed-period span coverage, and the next local midnight when the full calendar day becomes available. It automatically reads current.log and standard dated rotations. Log lines include their UTC offset, for example 2026-07-15 18:42:10,123 (UTC+3) APP_DEBUG ..., while the analyzer remains compatible with older lines that have no offset. Timestamp indexes, marker-routed event parsing, and fast ISO timestamp conversion keep large APP_DEBUG reports practical.

# After copying scripts/es-ess-daily-report.py to production
chmod +x /data/es-ESS/scripts/es-ess-daily-report.py
python -m py_compile /data/es-ESS/scripts/es-ess-daily-report.py

# In /data/es-ESS/config.ini
[Common]
LogLevel=APP_DEBUG
LogRetentionDays=10

# Restart, collect one complete day, then produce a human report
/data/es-ESS/restart.sh
python /data/es-ESS/scripts/es-ess-daily-report.py --date yesterday

# Optional partial report for diagnostic evidence available today
python /data/es-ESS/scripts/es-ess-daily-report.py --date today

# Private JSON report for retention or review
mkdir -p /data/es-ESS-validation
chmod 700 /data/es-ESS-validation
umask 077
python /data/es-ESS/scripts/es-ess-daily-report.py --date yesterday --json \
  > /data/es-ESS-validation/es-ess-daily-report-$(date +%Y%m%d).json

The report is read-only: it never writes D-Bus, MQTT, Wattpilot, configuration, files or service state. Optional current snapshots use only svstat and selected D-Bus GetValue calls. Interpret GOOD, ATTENTION, ANOMALY and INCOMPLETE using the complete es-ESS daily-report guide. The report covers runtime health, sanitized configuration, approximate charging sessions, phase/allowance/grace behavior, safety interventions and rare statuses 8–11 and 13–14. A rare status marked NOT OBSERVED is informational. Do not force grid import, telemetry failure, phase switching or battery assist merely to exercise a check.

Interactive execution shows progress for configuration, byte-level log loading, evidence checks, each read-only snapshot path, analysis and rendering. The progress bar is written to stderr, so JSON stdout remains valid. Use --no-progress for automation or --no-current-snapshot to skip the optional current snapshot; the required read-only Venus timezone query still runs. Snapshot calls time out after two seconds; after three consecutive D-Bus timeouts, remaining paths are marked unavailable and log analysis continues.

Do not place real passwords in screenshots or public documentation. Back up config.ini privately. Use kill_me.sh only when graceful restart fails; normal shutdown restores the configured grid-setpoint baseline and performs service cleanup.

10 · GX firmware

Upgrade deliberately and preserve a tested rollback

These steps cover a Cerbo GX upgrade to the official v3.75 release, followed by post-update checks and both firmware rollback methods. This es-ESS checkout supports only v3.75; booting older firmware also requires restoring an app checkout compatible with that firmware. The complete copyable runbook is also available in the Markdown firmware guide.

Maintenance window: do not update during active EV charging. Disconnect the vehicle or put Wattpilot in Manual mode, keep local access available, disable automatic firmware updates, and supervise the first Auto/Eco session after the upgrade.

SSH note: a Venus OS firmware update can reset the root password because the password is stored on the firmware root filesystem. After reboot, open Settings → Access & Security on the Cerbo console, enable Superuser access, confirm SSH on LAN is still enabled, and set a new root password if the previous one no longer works.

Before upgrading

  1. Record both firmware slots.
    Open Settings → General → Firmware. Record the currently running version, then open Stored backup firmware and record the version available to boot. Do not assume the stored slot is compatible with this es-ESS checkout.
  2. Confirm persistent modifications are enabled.
    Open Settings → General → Modification checks and confirm Modifications enabled. This allows the persistent /data/rc.local hook to restore the es-ESS service link after reboot.
  3. Back up es-ESS.
    Copy /data/es-ESS/config.ini to a private backup, confirm /data/es-ESS and /data/rc.local exist, and verify that /data/rc.local invokes /data/es-ESS/install.sh.
  4. Record system assignments.
    Save the ESS mode, grid setpoint, selected battery monitor, grid meter, PV AC position, MQTT settings and enabled es-ESS services. Verify Remote Console and SSH before proceeding.

Online upgrade to v3.75

  1. Open Settings → General → Firmware → Online updates and select the Official release feed.
  2. Select Check for updates and verify the offered firmware is the clean v3.75 release. The final official Cerbo GX image is build 20260624163305.
  3. Start installation, keep power connected, and wait for the Cerbo GX to reboot and reconnect.
  4. Return to Firmware and confirm v3.75 is running. Stored backup firmware should show the previous firmware slot; verify it explicitly.

Offline upgrade

  1. Download the official Cerbo GX venus-swu-einstein-...-v3.75.swu file and place it in the root of a FAT32 USB stick or microSD card.
  2. Insert it, open Settings → General → Firmware → Install from SD/USB, and select Check for updates on SD/USB.
  3. Verify that v3.75 was detected, start installation, wait for the reboot, remove the storage device, and verify both firmware slots.

Post-update checks

  1. Restore or confirm SSH access.
    On the Cerbo console, open Settings → Access & Security. Enable Superuser access, check that SSH on LAN is enabled, and set a new root password if password login no longer works. Do not store the root password in project files.

Rollback option 1: stored backup firmware

  1. Open Settings → General → Firmware → Stored backup firmware.
  2. Verify that the stored release is the firmware you intend to boot, then select Press to boot.
  3. After reboot, verify that the intended firmware is current. The former v3.75 installation becomes the stored option.
  4. Restore an es-ESS checkout compatible with the running firmware, rerun /data/es-ESS/install.sh, check Python dependencies, restart es-ESS, inspect logs, and repeat D-Bus, MQTT, Manual and supervised Auto/Eco checks.

Rollback option 2: manual firmware installation

  1. Download the intended official Cerbo GX .swu file from Victron's release archive.
  2. Place it in the root of a FAT32 USB stick or microSD card and insert it into the Cerbo GX.
  3. Open Settings → General → Firmware → Install from SD/USB, check for firmware, verify the detected clean image, and start installation.
  4. After reboot, verify cat /opt/victronenergy/version reports the intended firmware, restore an es-ESS checkout compatible with that firmware, restore the service link and dependencies if needed, and repeat the post-update validation.

Do not use factory reset for a normal rollback. The stored-firmware and manual .swu methods preserve /data in normal operation. A factory reset or full recovery image is a last-resort recovery action and can erase es-ESS and its configuration.

11 · Commissioning

Prove each safety boundary with live measurements

Pre-charge checks

  • Actual BMS is selected as the GX battery monitor.
  • Grid import/export signs agree on all three phases.
  • Battery discharge is negative in the system value used by the distributor.
  • Fronius PV production is visible and assigned to the correct AC position.
  • Wattpilot IP, password, reported voltage and effective current limit are valid.
  • SolarOverheadDistributor publishes zero allowance when a required input is missing.
  • With the vehicle disconnected, native Use PV surplus and flexible tariff are off and /CommandAuthorityOk=1 after Auto is selected from the VRM web dashboard EVCS tile/module.

Charging behavior

  • Manual/Standard charging is not altered by es-ESS.
  • Auto/Eco does not start below the one-phase allowance.
  • One-phase current follows allowance in one-ampere steps.
  • Three-phase switch occurs only after sustained threshold and cooldown.
  • A cloud starts bounded assist only after charging is already active.
  • Assist ends on SOC, power-shortfall or time limit.
  • Sustained grid import triggers phase-down or stop with grid charging disabled.
